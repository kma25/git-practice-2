Введение
Git — это распределённая система контроля версий. Она используется для сохранения истории изменений проекта, сравнения разных версий файлов, работы с ветками и восстановления предыдущего состояния проекта при необходимости.
Git особенно полезен при разработке программного обеспечения, написании документации и выполнении учебных проектов. С его помощью можно фиксировать каждый этап работы отдельным коммитом и видеть, какие изменения были внесены в проект.
GitHub — это онлайн-платформа для хранения Git-репозиториев. Она позволяет размещать проекты в интернете, работать с ветками, создавать pull request, просматривать историю изменений и организовывать совместную разработку.
В данной инструкции рассматриваются основные команды Git, работа с локальным и удалённым репозиторием, создание веток, слияние изменений и принцип работы с форками.

Git и GitHub: основные отличия
Git и GitHub часто используются вместе, но это разные инструменты.
Инструмент	Назначение
Git	Программа для контроля версий на локальном компьютере
GitHub	Веб-сервис для хранения репозиториев и совместной работы
Git Bash	Консольная оболочка для выполнения команд Git в Windows
Репозиторий	Хранилище проекта с историей изменений
Коммит	Зафиксированное состояние проекта
Ветка	Отдельная линия разработки
Главное отличие заключается в том, что Git работает локально на компьютере пользователя, а GitHub хранит удалённую копию репозитория в интернете.
Пример локальной команды Git:
```bash
git status
```
Пример ссылки на репозиторий GitHub:
```text
https://github.com/user/git-practice-2.git
```

Создание учётной записи GitHub
Для работы с удалёнными репозиториями необходимо создать учётную запись GitHub.
Порядок регистрации
Перейти на сайт GitHub.
Нажать кнопку `Sign up`.
Указать адрес электронной почты.
Придумать пароль.
Указать имя пользователя.
Подтвердить регистрацию через письмо на email.
Выполнить вход в аккаунт.
Создание нового репозитория
После регистрации можно создать новый репозиторий.
Порядок действий:
Нажать кнопку `New repository`.
Указать название репозитория.
Выбрать тип доступа: `Public` или `Private`.
При необходимости добавить описание.
Нажать кнопку `Create repository`.
Пример названия репозитория:
```text
git-practice-2
```
Пример ссылки на удалённый репозиторий:
```text
https://github.com/user/git-practice-2.git
```

Установка и первичная настройка Git
Для установки Git на Windows необходимо:
Перейти на официальный сайт Git.
Скачать установщик для Windows.
Запустить установочный файл.
На этапах установки оставить стандартные параметры.
После установки открыть Git Bash.
Проверить установку Git можно командой:
```bash
git --version
```
Пример результата:
```text
git version 2.45.0
```
Настройка имени пользователя
Имя пользователя будет отображаться в истории коммитов.
```bash
git config --global user.name "Имя пользователя"
```
Пример:
```bash
git config --global user.name "Student"
```
Настройка email
Email также сохраняется в информации о коммитах.
```bash
git config --global user.email "email@example.com"
```
Пример:
```bash
git config --global user.email "student@example.com"
```
Просмотр текущих настроек
```bash
git config --global --list
```
Пример:
```bash
git config --global --list
```

Основные состояния файлов в Git
Перед созданием коммита файлы проходят несколько состояний.
Состояние	Описание
Untracked	Файл создан, но ещё не отслеживается Git
Modified	Файл был изменён после последнего коммита
Staged	Файл добавлен в индекс и готов к коммиту
Committed	Изменения сохранены в истории репозитория
Общий процесс работы выглядит так:
```text
Изменение файла
↓
git add
↓
git commit
↓
git push
```
Команда для просмотра состояния файлов:
```bash
git status
```
Пример:
```bash
git status
```

Локальная работа с Git
Локальная работа выполняется на компьютере пользователя. На этом этапе создаётся репозиторий, добавляются файлы и фиксируются изменения.
Инициализация репозитория
Команда создаёт новый локальный Git-репозиторий в текущей папке.
```bash
git init
```
Пример:
```bash
git init
```
После выполнения команды в папке появляется скрытый каталог `.git`, в котором хранится история проекта.
Проверка состояния репозитория
Команда показывает изменённые, добавленные и неотслеживаемые файлы.
```bash
git status
```
Пример:
```bash
git status
```
Добавление файла в индекс
Команда добавляет конкретный файл в область подготовленных изменений.
```bash
git add имя_файла
```
Пример:
```bash
git add README.md
```
Добавление всех файлов
Команда добавляет все новые и изменённые файлы.
```bash
git add .
```
Пример:
```bash
git add .
```
Создание коммита
Коммит фиксирует изменения в истории репозитория.
```bash
git commit -m "Сообщение коммита"
```
Пример:
```bash
git commit -m "Добавлена инструкция по локальной работе с Git"
```
Сообщение коммита должно кратко объяснять, что было сделано.
Просмотр истории коммитов
Команда показывает историю изменений.
```bash
git log
```
Пример краткого вывода:
```bash
git log --oneline
```
Пример просмотра истории в виде графа:
```bash
git log --oneline --graph --all
```
Отмена изменений в файле
Команда отменяет несохранённые изменения в выбранном файле.
```bash
git checkout -- имя_файла
```
Пример:
```bash
git checkout -- README.md
```
Удаление файла из индекса
Команда убирает файл из индекса, но не удаляет его с компьютера.
```bash
git reset имя_файла
```
Пример:
```bash
git reset README.md
```

Работа с ветками
Ветка — это отдельная линия разработки. Ветки позволяют выполнять изменения независимо от основной версии проекта.
В рамках практической работы используются три ветки:
Ветка	Назначение
main	Основная ветка с итоговым вариантом инструкции
local	Ветка с описанием локальной работы с Git
global	Ветка с описанием удалённого Git и форков
Просмотр списка веток
```bash
git branch
```
Пример:
```bash
git branch
```
Текущая ветка отмечается символом `*`.
Создание ветки
```bash
git branch имя_ветки
```
Пример:
```bash
git branch local
```
Переключение на ветку
```bash
git checkout имя_ветки
```
Пример:
```bash
git checkout local
```
Также можно использовать команду:
```bash
git switch local
```
Создание ветки и переход в неё
```bash
git checkout -b имя_ветки
```
Пример:
```bash
git checkout -b global
```
Слияние веток
Команда объединяет изменения из выбранной ветки с текущей веткой.
```bash
git merge имя_ветки
```
Пример:
```bash
git checkout main
git merge local
```

Работа с удалённым репозиторием
Удалённый репозиторий хранится на GitHub или другом Git-сервисе. Он нужен для публикации проекта, резервного хранения файлов и совместной работы.
Клонирование репозитория
Команда создаёт локальную копию удалённого репозитория.
```bash
git clone ссылка_на_репозиторий
```
Пример:
```bash
git clone https://github.com/user/git-practice-2.git
```
Добавление удалённого репозитория
Команда связывает локальный репозиторий с удалённым.
```bash
git remote add origin ссылка_на_репозиторий
```
Пример:
```bash
git remote add origin https://github.com/user/git-practice-2.git
```
Просмотр удалённых репозиториев
```bash
git remote -v
```
Пример:
```bash
git remote -v
```
Отправка изменений на GitHub
```bash
git push origin имя_ветки
```
Пример:
```bash
git push origin main
```
Первая отправка ветки
Команда отправляет ветку на GitHub и связывает локальную ветку с удалённой.
```bash
git push -u origin имя_ветки
```
Пример:
```bash
git push -u origin local
```
После первой отправки можно использовать короткую команду:
```bash
git push
```
Получение изменений из удалённого репозитория
```bash
git pull origin имя_ветки
```
Пример:
```bash
git pull origin main
```
Загрузка изменений без автоматического слияния
```bash
git fetch
```
Пример:
```bash
git fetch origin
```

Pull Request и слияние изменений
Pull Request — это запрос на добавление изменений из одной ветки в другую. На GitHub он используется для просмотра изменений перед слиянием.
Обычно Pull Request создаётся после отправки отдельной ветки в удалённый репозиторий.
Пример работы с Pull Request
Создать ветку.
Внести изменения.
Сделать коммит.
Отправить ветку на GitHub.
Нажать `Compare & pull request`.
Создать Pull Request.
Проверить изменения.
Нажать `Merge pull request`.
Подтвердить слияние.
Пример команд перед созданием Pull Request:
```bash
git checkout -b local
git add README.md
git commit -m "Добавлены команды локального Git"
git push -u origin local
```
После слияния через GitHub нужно обновить локальную ветку `main`:
```bash
git checkout main
git pull origin main
```

Работа с форками
Форк — это копия чужого репозитория в своём аккаунте GitHub. Форки используются, когда нужно предложить изменения в чужой проект без прямого доступа к исходному репозиторию.
Принцип работы с форком
Открыть чужой репозиторий на GitHub.
Нажать кнопку `Fork`.
GitHub создаст копию репозитория в аккаунте пользователя.
Склонировать форк на компьютер.
Создать отдельную ветку.
Внести изменения.
Отправить изменения в свой форк.
Создать Pull Request в исходный репозиторий.
Клонирование форка
```bash
git clone ссылка_на_форк
```
Пример:
```bash
git clone https://github.com/user/project-fork.git
```
Добавление исходного репозитория
Исходный репозиторий обычно добавляется под именем `upstream`.
```bash
git remote add upstream ссылка_на_исходный_репозиторий
```
Пример:
```bash
git remote add upstream https://github.com/original-owner/project.git
```
Получение изменений из исходного репозитория
```bash
git fetch upstream
```
Пример:
```bash
git fetch upstream
```
Обновление своей ветки из исходного репозитория
```bash
git merge upstream/main
```
Пример:
```bash
git merge upstream/main
```
Отправка изменений в свой форк
```bash
git push origin имя_ветки
```
Пример:
```bash
git push origin fix-readme
```
После отправки изменений можно создать Pull Request в исходный репозиторий.
