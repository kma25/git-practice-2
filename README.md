## Введение

Git используется для контроля версий проекта. Он позволяет сохранять историю изменений, возвращаться к предыдущим версиям файлов, работать с ветками и фиксировать каждый этап разработки отдельным коммитом.

GitHub используется для хранения репозитория в интернете. Через GitHub можно отправлять изменения, сравнивать ветки, создавать Pull Request и работать над проектом совместно с другими пользователями.

В данной инструкции описаны основные действия при работе с Git и GitHub: создание учётной записи, установка Git, локальная работа с репозиторием, работа с ветками, удалённым репозиторием и форками.

## Основные понятия Git

| Понятие | Описание |
|---|---|
| Репозиторий | Папка проекта, в которой Git хранит историю изменений |
| Коммит | Зафиксированное состояние проекта |
| Ветка | Отдельная линия разработки |
| Индекс | Область подготовленных изменений перед коммитом |
| Удалённый репозиторий | Репозиторий, размещённый на GitHub или другом сервисе |
| Pull Request | Запрос на добавление изменений из одной ветки в другую |
| Fork | Копия чужого репозитория в своём аккаунте |

## Git и GitHub: в чём разница

Git и GitHub связаны между собой, но выполняют разные задачи.

| Инструмент | Назначение |
|---|---|
| Git | Работает локально на компьютере и хранит историю изменений |
| GitHub | Хранит удалённую копию репозитория в интернете |
| Git Bash | Консоль для выполнения команд Git в Windows |

Пример локальной команды Git:

```bash
git status
```

Пример ссылки на репозиторий GitHub:

```text
https://github.com/user/git-practice-2.git
```

## Создание учётной записи GitHub

Для работы с удалёнными репозиториями необходимо создать учётную запись GitHub.

Порядок действий:

1. Перейти на сайт GitHub.
2. Нажать кнопку `Sign up`.
3. Указать email.
4. Придумать пароль.
5. Указать имя пользователя.
6. Подтвердить регистрацию через письмо на почте.
7. Войти в созданный аккаунт.

После регистрации можно создать новый репозиторий.

Порядок создания репозитория:

1. Нажать кнопку `New repository`.
2. Указать имя репозитория.
3. Выбрать тип доступа: `Public` или `Private`.
4. Нажать кнопку `Create repository`.

Пример имени репозитория:

```text
git-practice-2
```

## Установка и настройка Git

Для установки Git на Windows необходимо скачать установщик Git, запустить его и оставить стандартные параметры установки. После установки работа с Git может выполняться через Git Bash.

Проверка версии Git:

```bash
git --version
```

Пример:

```text
git version 2.45.0
```

После установки нужно выполнить первичную настройку.

Настройка имени пользователя:

```bash
git config --global user.name "Student"
```

Настройка email:

```bash
git config --global user.email "student@example.com"
```

Проверка настроек:

```bash
git config --global --list
```

Эти данные будут отображаться в истории коммитов.

## Локальная работа с Git

Локальная работа выполняется на компьютере пользователя. Основной порядок действий выглядит так:

```text
создать или изменить файл
↓
добавить файл в индекс
↓
создать коммит
↓
при необходимости отправить изменения на GitHub
```

### Инициализация репозитория

```bash
git init
```

Команда создаёт локальный Git-репозиторий в текущей папке.

### Проверка состояния

```bash
git status
```

Команда показывает, какие файлы изменены, добавлены или ещё не отслеживаются Git.

### Добавление файла в индекс

```bash
git add README.md
```

### Добавление всех файлов

```bash
git add .
```

### Создание коммита

```bash
git commit -m "Добавлена инструкция по Git"
```

Сообщение коммита должно кратко объяснять, что было сделано.

### Просмотр истории

```bash
git log
```

Краткий вариант:

```bash
git log --oneline
```

Графическое отображение истории:

```bash
git log --oneline --graph --all
```

## Работа с ветками

Ветка позволяет вести отдельную линию разработки, не изменяя основную ветку сразу.

В данной работе используются три ветки:

| Ветка | Назначение |
|---|---|
| `main` | Основная ветка с итоговым вариантом инструкции |
| `local` | Ветка с описанием локальных команд Git |
| `global` | Ветка с описанием удалённого Git и форков |

### Просмотр веток

```bash
git branch
```

Текущая ветка отмечается символом `*`.

### Создание ветки

```bash
git branch local
```

### Создание ветки и переход в неё

```bash
git checkout -b local
```

### Переключение между ветками

```bash
git checkout main
```

Также можно использовать команду:

```bash
git switch local
```

### Слияние веток

```bash
git merge local
```

Пример:

```bash
git checkout main
git merge local
```

Сначала нужно перейти в ветку, в которую будут добавляться изменения, а затем выполнить `merge`.

## Работа с удалённым репозиторием

Удалённый репозиторий нужен для хранения проекта на GitHub.

### Клонирование репозитория

```bash
git clone ссылка_на_репозиторий
```

Пример:

```bash
git clone https://github.com/user/git-practice-2.git
```

### Подключение удалённого репозитория

```bash
git remote add origin ссылка_на_репозиторий
```

Пример:

```bash
git remote add origin https://github.com/user/git-practice-2.git
```

### Просмотр удалённых репозиториев

```bash
git remote -v
```

### Отправка изменений

```bash
git push origin main
```

### Первая отправка новой ветки

```bash
git push -u origin local
```

Пример:

```bash
git push -u origin global
```

### Получение изменений

```bash
git pull origin main
```

### Загрузка изменений без слияния

```bash
git fetch origin
```

## Pull Request и слияние изменений

Pull Request используется на GitHub для проверки изменений перед добавлением их в основную ветку.

Обычный порядок работы:

1. Создать отдельную ветку.
2. Изменить файл.
3. Сделать коммит.
4. Отправить ветку на GitHub.
5. Нажать `Compare & pull request`.
6. Создать Pull Request.
7. Проверить изменения.
8. Нажать `Merge pull request`.
9. Подтвердить слияние.

Пример команд перед созданием Pull Request:

```bash
git checkout -b local
git add README.md
git commit -m "Добавлены команды локального Git"
git push -u origin local
```

После слияния через GitHub нужно обновить локальную ветку `main`:

```bash
git checkout main
git pull origin main
```

## Работа с форками

Форк — это копия чужого репозитория в своём аккаунте GitHub. Он используется, когда нужно предложить изменения в чужой проект без прямого доступа к исходному репозиторию.

Порядок работы с форком:

1. Открыть чужой репозиторий на GitHub.
2. Нажать кнопку `Fork`.
3. Склонировать созданную копию на компьютер.
4. Создать отдельную ветку.
5. Внести изменения.
6. Отправить изменения в свой форк.
7. Создать Pull Request в исходный репозиторий.

### Клонирование форка

```bash
git clone ссылка_на_форк
```

Пример:

```bash
git clone https://github.com/user/project-fork.git
```

### Добавление исходного репозитория

Исходный репозиторий обычно добавляется под именем `upstream`.

```bash
git remote add upstream ссылка_на_исходный_репозиторий
```

Пример:

```bash
git remote add upstream https://github.com/original-owner/project.git
```

### Получение изменений из исходного репозитория

```bash
git fetch upstream
```

### Обновление своей ветки

```bash
git merge upstream/main
```

### Отправка изменений в свой форк

```bash
git push origin имя_ветки
```

Пример:

```bash
git push origin fix-readme
```
